1. Purpose & Context

This BRD captures the business requirements implied by the currently implemented system. The platform consolidates brokerage holdings and converts them into an ownership-weighted view of company financial statements (balance sheet and income statement), with a billing/paywall model and password lifecycle support.

1.1 Business Objectives

  • Provide a unified dashboard for multi-brokerage holdings.
  • Offer institutional-style look-through analysis of owned companies.
  • Support both direct brokerage connectivity and manual CSV upload fallback.
  • Enable conversion via paid subscription plans and account activation.

2. Scope

2.1 In Scope (As Implemented)

  • Brokerage connect UX with direct SnapTrade and manual upload alternatives.
  • Holdings ingestion, normalization, persistence, and display.
  • Balance sheet and income statement aggregation from holdings + financial datasets.
  • Login/logout flows, password reset and set-password flows.
  • Billing page with Stripe pricing table and Stripe webhook subscription processing.

2.2 Out of Scope (Current Build)

  • Trade execution, transfers, or brokerage-side write operations.
  • Role-based access control and route-level authorization enforcement.
  • Native mobile app support.

4. Functional Requirements (FR)

FR-A: Access, Login, and Session

IDRequirementAcceptance Criteria
FR-A1 The application shall default users to /brokeragesAndAccounts. Root and unknown routes resolve to brokerages page.
FR-A2 The system shall authenticate login via public.users password hash comparison. Valid credentials create local user session state in auth store.
FR-A3 The system shall enforce subscription recency checks at login. Expired subscriptions are denied with renew CTA and no session set.
FR-A4 The system shall persist auth context in browser local storage. auth-storage retains user + SnapTrade transient context.
FR-A5 The system shall provide logout actions from multiple UI entry points. Sidenav sign out clears auth and app cache; navbar logout clears auth only (as-built).

FR-B: Password Lifecycle

IDRequirementAcceptance Criteria
FR-B1 The system shall allow users to request a password setup/reset email. Valid email request triggers server-side reset-link generation and email send flow.
FR-B2 The system shall generate time-limited password tokens. Tokens stored/upserted with 30-minute expiry in password_reset_tokens.
FR-B3 The system shall support password set by token. Valid token updates users.password_hash; used token deleted.
FR-B4 The system shall support activation mode messaging. mode=activation changes page copy while using same set-password backend.

FR-C: Brokerage Connectivity (SnapTrade + Manual)

IDRequirementAcceptance Criteria
FR-C1 The system shall present supported brokerages by country. User can select Canada/United States and choose a brokerage tile.
FR-C2 The system shall determine direct vs manual path from integration availability, feature flag, and allowlist. Unsupported/disabled/unallowlisted brokerages open manual upload modal.
FR-C3 The system shall create transient SnapTrade user context for direct connect. Generates transient user ID, obtains user secret, requests login link, opens portal iframe.
FR-C4 The system shall process SnapTrade completion events. Success event redirects to sync route; errors trigger fallback/manual guidance.
FR-C5 The system shall provide timed fallback to manual upload when direct connect does not initialize quickly. Fallback UI appears after timeout with template guidance and upload controls.

FR-D: Manual CSV Ingestion

IDRequirementAcceptance Criteria
FR-D1 The system shall parse broker CSVs into account metadata and holdings. Parses first table key-values and second table Symbol/Market/Quantity rows.
FR-D2 The system shall normalize and store parsed account holdings. Writes to brokerages/accounts slices and per-account holdings maps.

FR-E: Statement Analytics

IDRequirementAcceptance Criteria
FR-E1 The system shall aggregate balance sheet and income statement metrics per holding. Uses latest financial tag values and share-based prorating formula.
FR-E2 The system shall compute ownership-share representation. Displays ratio format (e.g., 1 in X) when shares outstanding is available.
FR-E3 The system shall keep rows visible with N/A when data is partial/unavailable. No silent dropping of companies solely due to missing tags.
FR-E4 The system shall enforce market availability constraints for account participation. Only NASDAQ/NYSE holdings count as supported in analysis selectors.
FR-E5 The system shall provide PDF export of current statement view. Export action includes title/subtitle/account label/rows in generated PDF.

FR-F: Monetization & Provisioning

IDRequirementAcceptance Criteria
FR-F1 The system shall display Stripe pricing table in Billing page. Pricing table renders via configured pricing table and publishable key.
FR-F2 The system shall process successful Stripe invoice events. Webhook validates signature and handles invoice.payment_succeeded.
FR-F3 The system shall upsert user subscription profile by customer email. Writes/updates email, phone, interval, and payment timestamp in users table.
FR-F4 The system shall send activation email after successful subscription processing. Email contains tokenized link with activation mode parameter.

5. Workflow Definitions

5.1 Brokerage Connection Workflow

  1. User opens Connect Brokerage dialog.
  2. System evaluates direct/manual eligibility.
  3. If direct: register transient user, get portal URL, launch iframe, await completion events.
  4. On success: redirect to sync route and fetch account data from backend.
  5. Map and persist account/holdings slices; timestamp connection.
  6. If direct fails/times out: display manual upload fallback.

5.2 Statement Generation Workflow

  1. Resolve selected/default account and holdings set.
  2. For each symbol, retrieve financial rows.
  3. Pick latest values by required tags.
  4. Compute prorated metrics per position quantity.
  5. Render rows with paywall behavior based on login state.

5.3 Password Reset Workflow

  1. User requests reset link with email.
  2. Backend creates/upserts token and sends email.
  3. User opens tokenized link and submits new password.
  4. Backend validates token, updates password hash, removes token.

8. Assumptions, Dependencies, and Constraints

8.1 Assumptions

  • Supabase endpoints and edge function secrets are correctly provisioned.
  • SnapTrade credentials and broker slug mappings are valid for target environment.
  • Stripe pricing table and webhook secrets are configured.

8.2 External Dependencies

  • Supabase (REST + Edge Functions)
  • SnapTrade API + portal messaging
  • Stripe pricing table + webhooks
  • Postmark email delivery

8.3 Constraints

  • Current market-support gating only includes NYSE/NASDAQ in statement flows.
  • Current login model is custom table-based, not standard Supabase Auth sign-in.
  • localStorage is an active persistence mechanism for account and session context.

9. Known Gaps / As-Built Risks

  • Logout inconsistency: Sidenav Sign Out clears app cache; top-navbar Logout does not.
  • Production logging verbosity: Multiple client-side console.log debug statements remain in core user flows (brokerage, set-password, statement aggregation), increasing noise and potential data exposure in browser logs.
  • Security posture: encryption key and salt are hardcoded client-side for password crypto workflow.
  • Auth model divergence: app mixes custom user-table login with Supabase auth event listeners.

10. Success Criteria (Business)

  • Users can connect at least one brokerage (direct or manual fallback) and see holdings in dashboard.
  • Users can view prorated financial statements derived from holdings with predictable paywall behavior.
  • Paid subscription events provision or update user account records and trigger activation path.
  • Password lifecycle (request link -> set password) completes reliably within token TTL.
